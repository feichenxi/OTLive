## 一、概述

OTLive系统的语音对讲功能采用Windows PC + ESP32-S3的架构，通过WiFi网络实现实时音频传输。

### 3.3 音频缓存机制

ESP32-S3采用多级音频缓存机制，确保音频播放的连续性和稳定性：

#### 3.3.1 DMA缓冲区（I2S硬件层）

| 参数 | 值 | 说明 |
|------|-----|------|
| DMA缓冲区数量 | 64 | `dma_buf_count = 64` |
| 每个缓冲区样本数 | 512 | `dma_buf_len = 512` |
| 总缓冲区大小 | 64KB | 64 × 512 × 2字节 = 65536字节 |
| 缓冲区时长 | ~2秒 | 64 × 512 ÷ 32000 = 1.024秒 |

**DMA缓冲区工作原理：**
- I2S外设通过DMA方式从内存缓冲区读取音频数据
- 64个缓冲区形成环形队列，循环使用
- 当一个缓冲区播放完毕时，自动切换到下一个缓冲区
- 不需要CPU持续干预，降低CPU占用

#### 3.3.2 应用层缓冲区

| 参数 | 值 | 说明 |
|------|-----|------|
| 单次写入样本数 | 512 | `i2s_write` 每次写入512样本 |
| 写入超时 | 无限等待 | `portMAX_DELAY` 表示如果DMA缓冲区满则等待 |

**应用层缓存策略：**
- 接收到UDP数据包后，立即写入I2S DMA缓冲区
- `i2s_write` 使用 `portMAX_DELAY` 确保数据一定被写入
- 即使网络暂时抖动，DMA缓冲区中的音频仍可连续播放

#### 3.3.3 缓存机制优势

1. **消除断续问题**：64KB的总缓冲区可容纳约1秒的音频数据，网络短暂抖动不会影响播放
2. **降低延迟**：采用流水线式处理，数据到达后立即写入缓冲区开始播放
3. **稳定播放**：DMA硬件自动处理缓冲区切换，避免播放中断
4. **CPU效率**：DMA方式播放，CPU只需负责接收数据写入缓冲区

#### 3.3.4 代码实现

**I2S DMA配置（main.cpp:991-1025）：**
```cpp
i2s_config_t i2s_config = {
    .mode = (i2s_mode_t)(I2S_MODE_MASTER | I2S_MODE_TX),
    .sample_rate = SAMPLE_RATE,
    .bits_per_sample = BITS_PER_SAMPLE,
    .channel_format = I2S_CHANNEL_FMT_ONLY_LEFT,
    .communication_format = I2S_COMM_FORMAT_I2S,
    .intr_alloc_flags = ESP_INTR_FLAG_LEVEL1,
    .dma_buf_count = 64,              // 64个DMA缓冲区
    .dma_buf_len = 512,               // 每个缓冲区512样本
    .use_apll = false,
    .tx_desc_auto_clear = true,
    .fixed_mclk = 0
};

i2s_driver_install(I2S_NUM, &i2s_config, 0, NULL);
i2s_zero_dma_buffer(I2S_NUM);        // 清零DMA缓冲区
```

**音频数据写入（main.cpp:380）：**
```cpp
// 直接写入I2S，使用portMAX_DELAY确保写入成功
i2s_write(I2S_NUM, audioData, packetSize, &bytes_written, portMAX_DELAY);
```

## 七、音量控制

### 7.1 Windows端音量控制

- **对讲音量**：每个房间独立配置，范围0-100
- **应用方式**：在发送前对音频样本应用音量增益
- **增益计算**：`gain = volume / 100.0`
- **防止溢出**：`sample = max(-32768, min(32767, sample * gain))`

### 7.2 ESP32-S3端音量控制

- **主音量**：全局音量，范围0.0-1.0
- **默认值**：0.5
- **应用位置**：I2S输出前对音频数据缩放

## 八、音频电平显示

### 8.1 RMS计算

Windows端定期计算音频RMS（均方根）值并发送到前端显示：

```python
rms = sqrt(avg(sample²))
level = (rms / 32767) * 100
```

### 8.2 性能优化

- 只使用部分样本计算（前200个样本）
- 每20个音频块更新一次电平
- 通过WebSocket发送到前端

## 十、开发注意事项

### 10.1 采样率一致性

- Windows和ESP32-S3必须使用相同的采样率（48000Hz）
- 如果采样率不匹配，会导致播放速度异常或失真

### 10.2 网络延迟

- UDP协议无重传机制，丢包会导致音频卡顿
- 网络延迟过高会影响实时性
- 建议使用5GHz WiFi或有线网络

### 10.3 缓冲区大小

- Windows端发送缓冲区：1024字节（512样本）
- ESP32-S3接收缓冲区：512样本
- 过小会导致频繁发送，过大增加延迟

### 10.4 数据格式

- 小端序（Little Endian）
- 16位有符号整数（int16_t）
- 单声道数据排列

### 10.5 超时机制

- ESP32-S3 30秒无数据自动关闭对讲
- 避免长时间占用音频资源
- 防止异常情况下的死锁

## 十二、AI语音播放时序控制（特别重要）

### 12.1 问题背景

OTLive系统支持两种AI语音生成方式：
- **非流式**：先生成完整WAV文件，再播放
- **流式**：边生成边播放

**关键问题**：流式播放时出现严重噪音，而非流式播放完全正常。

### 12.2 问题根因分析

两种方式的时序控制差异：

| 方式 | 时序控制方法 | 时间精度 | 结果 |
|------|-------------|---------|------|
| 非流式 | `time.perf_counter()` + 绝对时间基准 + 忙等待 | 微秒级 | ✅ 正常 |
| 流式（原代码） | `time.sleep(packet_duration * 0.98)` | 10-15ms（Windows） | ❌ 噪音 |

**核心问题**：
- Windows下 `time.sleep()` 精度极低，通常只有10-15ms
- 使用相对时间控制，时序漂移严重
- 导致音频数据包发送间隔不均匀，产生噪音

### 12.3 解决方案

**统一使用精确时序控制**，流式和非流式采用完全相同的时序算法：

```python
def _high_precision_sleep(self, duration: float):
    end_time = time.perf_counter() + duration
    while time.perf_counter() < end_time:
        pass

# 播放时序控制
start_time = time.perf_counter()  # 绝对时间基准
next_send_time = start_time

for each_packet:
    current_time = time.perf_counter()
    if current_time < next_send_time:
        self._high_precision_sleep(next_send_time - current_time)
    
    send_packet()
    
    # 计算下一个发送时间点（绝对时间）
    next_send_time = start_time + (chunk_count * packet_duration)
```

### 12.4 关键改进点

1. **时间源**：使用 `time.perf_counter()`，Windows下精度可达微秒级
2. **绝对时间基准**：建立 `start_time`，所有发送时间基于此计算
3. **精确发送时间点**：`next_send_time = start_time + (chunk_count * packet_duration)`
4. **忙等待**：使用 `_high_precision_sleep()` 替代 `time.sleep()`
5. **时序偏移监控**：每50个包输出一次偏移量，便于调试

### 12.5 验证结果

- ✅ 流式播放噪音完全消除
- ✅ 音质与非流式完全一致
- ✅ 时序偏移控制在±1ms以内
- ✅ 保留流式低延迟特性

### 12.6 代码位置

- 时序控制实现：`iot/core/audio.py:AIVoicePlayer._high_precision_sleep()`
- 流式播放方法：`iot/core/audio.py:AIVoicePlayer.play_streamed_voice()`
- 非流式播放方法：`ai/aiwav/__init__.py:GiftVoiceGenerator._send_wav_to_esp32()`

**⚠️ 重要提醒**：任何音频播放到ESP32的代码，都必须使用此精确时序控制，否则会产生噪音！
